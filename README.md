# ERP-система Московского зоопарка
Программа моделирует автоматизацию основных процессов зоопарка: учет животных, распределение по вольерам, работу сотрудников (смотрителей, ветеринаров), организацию ухода (кормление, лечение, уборку), проведение выставок и формирование отчетности.

## Реализация принципов SOLID

### 1. Single Responsibility Principle (SRP)
- Каждый класс имеет одну ответственность:
  - `Animal` и его наследники - хранение данных о животных
  - `Zookeeper`, `Veterinarian` - выполнение обязанностей сотрудников
  - `FeedingService`, `MedicalService` - реализация конкретных услуг
  - `ReportModule` - генерация отчетов
  - `Enclosure` - управление вольерами

### 2. Open/Closed Principle (OCP)
- Система открыта для расширения, но закрыта для изменений:
  - Можно добавить новый тип животного (например, `Elephant extends Animal`)
  - Можно добавить новый сервис ухода (реализовать `AnimalCare`)
  - Можно добавить нового сотрудника (унаследовать от `Employee`)
  - Без изменения существующего кода

### 3. Liskov Substitution Principle (LSP)
- Все наследники `Animal` могут быть использованы вместо родителя:
  - `Lion`, `Parrot`, `Snake`, `RobotAnimal` взаимозаменяемы в списках животных
  - Все животные проходят базовую регистрацию в системе
  - Демонстрация в коде: `List<Animal> animals` содержит разные типы

### 4. Interface Segregation Principle (ISP)
- Интерфейсы разделены по функциональности:
  - `Feedable` - только для животных, которых нужно кормить
  - `MedicalCheckable` - только для животных, требующих медосмотра
  - `Exhibitable` - только для животных, участвующих в выставках
  - Пример: `RobotAnimal` не реализует `Feedable` и `MedicalCheckable`

### 5. Dependency Inversion Principle (DIP)
- Зависимости от абстракций, а не от конкретных реализаций:
  - `Employee` зависит от `AnimalCare` (интерфейс), а не от конкретных сервисов
  - `ReportModule` работает с интерфейсами `Feedable`, `MedicalCheckable`
  - `Exhibition` зависит от `Exhibitable`, а не от конкретных животных

## Потенциальные проблемы при расширении

1. **Добавление сложных правил совместимости животных** - потребуется изменить логику `Enclosure`
2. **Введение иерархии вольеров** - текущая плоская структура может не подойти
3. **Интеграция с внешними системами** (касса, онлайн-билеты) - потребуются адаптеры
4. **Сложные бизнес-правила кормления** - может потребоваться переработка `FeedingService`

## Преимущества введенных абстракций

1. **Гибкость** - легко добавлять новые виды животных и сервисы
2. **Тестируемость** - можно мокать интерфейсы при тестировании
3. **Читаемость** - четкое разделение обязанностей
4. **Сопровождаемость** - изменения в одном модуле минимально влияют на другие
5. **Масштабируемость** - система готова к добавлению нового функционала
